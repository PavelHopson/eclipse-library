# Token-saving agent stack для Codex / Claude

> Рабочий стандарт Eclipse Forge: экономим токены не “магией”, а дисциплиной контекста.
> Цель — меньше повторных чтений, меньше лишних инструкций, меньше бесполезного кода,
> больше проверяемого результата.

## Что применяем по умолчанию

| Слой | Инструмент / подход | Как используем |
|---|---|---|
| Минимальный код | [Ponytail](https://github.com/DietrichGebert/ponytail) | Перед реализацией проверяем: можно ли сделать меньше, есть ли stdlib/native API, нужен ли новый dependency, можно ли не трогать архитектуру |
| Контекстная дисциплина | [Context Engineering Skills](https://github.com/muratcankoylan/agent-skills-for-context-engineering) | Читаем только релевантные файлы, передаём точные пути/строки, не льём в модель целые логи и README без необходимости |
| Сжатые внутренние циклы | [caveman](https://github.com/JuliusBrussee/caveman) | Только для внутренних agent loops, черновых summaries и cost-control. Не включать для финального клиентского текста, UI-copy и точных объяснений |
| Дедуп контекста | [sqz](https://github.com/ojuschugh1/sqz) | Кандидат на тест: shell hook / MCP / IDE layer, чтобы один и тот же файл не отправлялся в модель несколько раз. Сначала sandbox, потом решение по внедрению |
| Аудит skills/MCP | [Loopy](https://github.com/Forward-Future/loopy) | Перед массовой установкой skills/MCP ищем дубли, конфликтующие инструкции, risky tools и автоматизации, которые гоняют агент по кругу |
| Память вместо повторов | `AGENTS.md` / `MEMORY.md` / agentmemory / claude-mem | Держим стабильные правила в файлах памяти, а не пересказываем их каждый раз. Внешние memory-plugins — только project-scoped и без секретов |
| Provider routing | OmniRoute / ClawRouter | Для продуктов и агентов с API: дешёвые модели на простое, сильные — на сложное, fallback и cost metrics. Только легальные ключи и ToS-safe режим |

## Правила для наших сессий

1. Не читать весь репозиторий, если задача решается 3–5 файлами.
2. Не пересказывать пользователю длинные tool outputs, если нужен только вывод.
3. Не добавлять dependency, если задача решается стандартными средствами.
4. Не писать “универсальную платформу”, если пользователь просит один сценарий.
5. Не держать в prompt повторяющиеся правила: правило должно жить в `AGENTS.md`, `MEMORY.md` или project docs.
6. Для длинной задачи сначала сделать короткую карту контекста, потом работать итерациями.
7. Если файл уже читали, не читать его повторно без причины — использовать точечный `rg` / line window.
8. После выполнения фиксировать решения в roadmap/docs, чтобы следующий агент не платил токенами за повторное восстановление контекста.

## Что НЕ делаем

- Не ставим все token-saving tools глобально без аудита.
- Не используем lossy-сжатие (`pxpipe`) для кода, секретов, чисел, юридических текстов и миграций.
- Не включаем caveman-режим в пользовательские отчёты, где важны стиль, доверие и точность.
- Не доверяем маркетинговым заявлениям “−92% токенов” без собственного benchmark.
- Не даём memory-плагинам доступ к `.env`, ключам, приватным логам и клиентским данным без явного allowlist.

## Минимальный benchmark перед внедрением sqz / memory plugins

1. Выбрать 3 типовые задачи: маленький fix, средний UI change, большой repo audit.
2. Прогнать без tool и с tool.
3. Сравнить: total tokens, повторные file reads, wall-clock time, количество ошибок, качество результата.
4. Проверить, не ломает ли tool apply_patch, git diff, tests и security boundaries.
5. Решить: global default, project-only или reference-only.

## Рекомендуемый порядок внедрения

1. **Сразу в правила работы:** Ponytail mindset + Context Engineering discipline.
2. **Для внутренних loops:** caveman как режим “коротко и по делу”.
3. **Перед установкой новых MCP/skills:** Loopy-аудит.
4. **На sandbox:** sqz как dedup-layer.
5. **Для продуктов:** OmniRoute/ClawRouter только там, где есть измеримый выигрыш по стоимости/latency.

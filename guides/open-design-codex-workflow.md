# 🎨 Open Design + Codex: шпаргалка по генерации сайта

> Практический сценарий: как развернуть Open Design внутри Codex, получить первый дизайн сайта на канве и затем улучшать его точечными правками, не переписывая весь проект с нуля.

Open Design полезен как промежуточный слой между “промпт в пустоту” и нормальной разработкой. Он даёт канву, артефакт, файлы и быстрый итеративный цикл: сгенерировал блок → посмотрел → поправил конкретный участок → перенёс в React / Next.js / Tailwind.

## Когда использовать

- Нужно быстро получить первый визуальный драфт лендинга, SaaS-страницы, портфолио или продуктового экрана.
- Нужен не просто код, а дизайн-направление: композиция, стиль, секции, визуальные акценты.
- Хочется дорабатывать блоки итеративно: hero отдельно, карточки отдельно, CTA отдельно.
- Нужно подготовить handoff в Eclipse Forge проекты: `EclipseForgeLanding`, лендинг Eclipse Chat, kwork-страницы, клиентские one-page продукты.

## Шаг 1. Автоматическая установка Open Design

Открой новый чат в Codex и отправь агенту:

```text
Помоги мне развернуть локально и запустить Open Design внутри Codex: https://github.com/nexu-io/open-design
Склонируй репозиторий, установи зависимости, запусти локальное приложение и открой его во встроенном браузере. Если возникнут ошибки — объясняй их простыми словами и исправляй.
```

Что должен сделать агент:

- скачать Open Design;
- установить Node.js зависимости;
- поднять локальный сервер;
- открыть приложение во встроенном браузере Codex;
- объяснить и исправить ошибки версий или зависимостей, если они появятся.

## Шаг 2. Фикс ошибки авторизации 401

Иногда первая генерация падает с `401 Unauthorized`: Open Design не видит подключение к агентскому движку или ключам.

В этом случае отправь:

```text
Сделай подключение opendesign к codex cli
```

Важно: не складывай токены и ключи в публичные файлы репозитория. Если агент просит секреты, держи их в `.env`, локальном конфиге или в безопасном хранилище, которое не попадает в Git.

## Шаг 3. Первая генерация сайта

Когда откроется интерфейс Open Design с канвой, дай задачу на первый драфт.

Пример промпта:

```text
С помощью OpenDesign спроектируй dark-mode лендинг для сервиса, который помогает выбрать и купить AI-подписки: ChatGPT, Claude, Gemini, Grok, Cursor и другие.
Цель: объяснить, что сервис помогает подобрать ИИ без переплат.
Нужны: современный AI/SaaS стиль, тёмная тема, аккуратные градиенты, крупный hero-блок, карточки подписок, блок сравнения, CTA в Telegram. Без перегруза и кислотных цветов. Сделай первый вариант дизайна на канве.
```

На выходе должен появиться HTML/CSS/JS артефакт с предпросмотром.

## Шаг 4. Итеративная доработка

Главное правило: не просить “сделай всё лучше”. Лучше указывать конкретный блок и конкретную цель.

### Разгрузить hero

```text
Сделай hero-блок менее перегруженным. Нужно: меньше текста; больше воздуха; более сильный заголовок; заметный CTA в Telegram; сохранить общий dark SaaS стиль. Не меняй всю страницу, работай только с hero-блоком.
```

### Усилить карточки

```text
Карточки подписок сделай более контрастными и понятными. Для каждой карточки должны быть: название AI-сервиса; для кого подходит; короткое преимущество; визуальный акцент (цветная боковая линия или мягкий glow).
```

### Добавить новую секцию

```text
Добавь новую секцию "Какой AI кому подходит".
Категории: для работы; для кода; для учебы; для изображений; для общения и поиска. Секция должна выглядеть частью текущего дизайна.
```

## Полезные приёмы

- **Итеративность:** проси 2-3 варианта одного блока, если направление пока не попало.
- **Точечность:** формулируй “работай только с hero/cards/CTA/contact”, чтобы агент не ломал композицию.
- **Стиль:** вместо “сделай красиво” давай визуальный язык: Linear, Vercel, OpenAI, Raycast, premium dark SaaS, editorial, cinematic.
- **Handoff:** когда дизайн готов, попроси перенести структуру в React / Next.js / Tailwind и сохранить визуальную иерархию.
- **Контроль качества:** после генерации прогоняй mobile, читаемость, скорость, контраст и реальные CTA.

## Куда применяем в Eclipse

| Проект | Как использовать |
|---|---|
| EclipseForgeLanding | Быстрые варианты секций, промо-блоки проектов, kwork entry-points |
| Eclipse Chat | Лендинг, onboarding, pricing/feature pages, product screenshots storytelling |
| Eclipse AI Hub | UI-концепты модулей: RAG, Arena, Copywriter, Image Studio |
| Shotforge / Text2Image | Визуальные лендинги под генеративные сервисы и примеры output |
| Клиентские лендинги | Первый дизайн-драфт перед переносом в production-стек |

## Ограничение

Open Design не заменяет финальный frontend-review. После handoff всё равно проверяем:

- адаптив;
- доступность и контраст;
- SEO / Open Graph;
- формы и реальные CTA;
- отсутствие лишних зависимостей;
- сборку production-проекта.

Правильная роль Open Design: быстро дать сильный визуальный первый слой, который затем инженерно доводится в основном репозитории.
